Track finding is one of the key steps in track reconstruction. It is responsible for identifying hits which belong to the same particles. A perfect track finding algorithm can associate all hits to unique tracks which correspond to the particles that traversed the detector. In reality, however, there will be thresholds which will prevent the algorithm from finding all tracks. Particles might create too few hits due to decy, their momentum being too small or being stopped by the material they traverse. This will result in hits which cannot be associated to a track. At the same time, detectors are not perfect and will have dead modules which results in missing hits, and noise which will create fake hits. These additional hits can result in tracks which were not created by a single particle, so called fake tracks. In summary, a good track finding algorithm needs to be able to deal with these issues and find as many tracks as possible while keeping the number of fake tracks as low as possible.

Associating hits to tracks is a combinatorial problem. The number of possible combinations grows superlinearly with the number of hits. For busy events with many hits, the track finding step in a reconstruction chain is usually the most expensive one in terms of compute time. Therefore, it has to be well optimized and balanced between physics and computational performance.

ACTS provides a Combinatorial Kalman Filter (CKF) implementation which can be used as a component for the track finding step in an experiment. The CKF depends on an initial estimate of the track parameters, which can be obtained from a prior seeding step. It searches iteratively for compatible measurements and updates the track parameters and their covariance using the Kalman filter formalism. Given measurement ambiguity, the CKF can branch into multiple hypotheses.

While ACTS already had a working prototype implementation for the CKF, it was effectively rewritten during the work on this thesis to improve its performance, flexibility, and maintainability. The new implementation and the most important changes are presented in this chapter. First, the CKF algorithm and its implementation is described in detail. Then, the refactor of the CKF into a track finding component is discussed. A two-way track finding strategy is introduced which allows to recover all measurements independent from the seed position. The branch stopper extension is presented which allows to stop the track search early. Modeling of branches with the track EDM is discussed and the branch finalization is explained. Two techniques to improve the track finding performance are presented: seed deduplication and stay-on-seed strategy. Finally, the chapter is concluded with a discussion of the initial covariance and the shortcomings of the CKF implementation.

\section{The CombinatorialKalmanFilter}

\begin{itemize}
    \color{red}
    \item depth first search
\end{itemize}

The \texttt{CombinatorialKalmanFilter} in ACTS is a Core algorithm that uses initial track parameters to find compatible measurements and fit them on a track.  As such, it heavily relies on the propagation of track parameters and their covariance through the tracking geometry and magnetic field to find all the sensitive surfaces that the track traverses. When the CKF arrives on a surface, it calls the \texttt{TrackStateCreator} to retrieve all compatible measurements given the predicted track parameters and covariance on the surface. The \texttt{TrackStateCreator} is provided by the experiment and an interface for ACTS to access the measurements. This step might also perform a calibration of the measurement given the track hypotheses. The newly created track states are then used to update the track parameters and their covariance using the Kalman filter formalism. In case no compatible measurements are found, the hole count of the track is increased. If material is encountered, the CKF will calculate the mean energy loss and multiple scattering and update the track parameters and their covariance. As a final step, the \texttt{BranchStopper} is called to decide whether the CKF should continue or stop the search. This procedure is repeated until the CKF reaches the end of the tracking geometry, the path limit is reached, or the branch stopper decides to stop. The resulting tracks are then stored and returned to the user.

Given that the Kalman filter is optimal for linear systems, the CKF has an optimal discrimination power for selecting measurements in this case. Measurements are usually selected based on their compatibility with the track parameters using a $\chi^2$ test. Given the degrees of freedom, the user effectively selects a p-value threshold to decide whether a measurement is compatible with the track.

The CKF will provide optimal resolution on the last track state that has a measurement. A smoothing pass can be performed, to inform the other track states about all found measurements. This results in fully fitted tracks with optimal resolution, while users may want to refit tracks given an estimated particle hypothesis.

The usage of the \texttt{CombinatorialKalmanFilter} is exercised in the Examples in the \texttt{TrackFindingAlgorithm}. Given a configuration and estimated track parameters, it will try to find all tracks and store them as intermediate event data. Another algorithm can then refit these tracks or perform a vertex search.

\section{CKF as a component}

The \texttt{CombinatorialKalmanFilter} used to be a monolithic algorithm with multiple responsibilities. It it used to deal with finding compatible measurements, fitting the track parameters, smoothing the track, and extrapolating the track to a reference surface. The code was hard to reason about and the error handling was not done properly. For instance, a problem during the smoothing or extrapolation step would silently affect the finding efficiency.

An important missing piece to the CKF was the ability to find measurements in both directions. The CKF was only able to find measurements in the particles' momentum direction. Since seeds are not required to be formed on the innermost layer of the detector, measurements on such inner layers might be missed which results in degredated efficiency and resolution. Adding this feature to the CKF would have further complicated the code and made it even harder to debug and maintain.

To address these issues, the \texttt{CombinatorialKalmanFilter} has been reduced to a pure track finding component. Smoothing and extrapolation have been extracted to toolbox functions, a track finding algorithm can call if necessary and at the right time. This way, the experiment is in control of the track finding process and can directly steer when to spend precious compute time.

\section{Two-way track finding}

\begin{itemize}
    \color{red}
    \item CKF only branches in one direction
\end{itemize}

In order to find all measurements a particle has left in the detector, the CKF either needs to start at the very first measurement or needs to be able to look for measurements in both directions.

After decoupling the CKF from the smoothing and extrapolation, it can simply be used to run twice. Once with the initial track parameters from the seed, and a second time in reversed direction with the smoothed parameters at the first measurement. This has first beed implemented and validated with the \texttt{TrackFindingAlgorithm} in the ACTS Examples, and later in Athena.

The remaining challenge is to stitch the resulting track segments together. The ACTS track EDM does not dictate a preferred direction for the stored tracks. But it does also not have a way to express the direction of a track. Given that, tracks should always be stored with a consistent direction. For this reason, track segments pointing in the opposite direction need to be reversed before they are concatenated to a full track. This involves an inverse of the bound-to-bound Jacobian which is stored on each track state and used for the smoothing.

\section{Branch stopper}

Seeding algorithms can produce many seeds which will not result in a track. The CKF will then spend compute time to propagate through the tracking geometry, only to find out that the track has too many holes and too few measurements.

A \texttt{BranchStopper} can be used to stop the CKF early if it is clear that the track will not be found. The \texttt{BranchStopper} is a user provided function which can observe the current track and its track states to make its decision. There are three possible outcomes: continue, stop and drop, and stop and keep. The first option will continue the CKF as usual. The second option will stop the CKF and drop the track. The third option will stop the CKF and keep the track.

\section{Track container for branches}

Branches discovered during running the CKF are now stored as tracks in a track container. The reason for this is that these CKF branches ressemble tracks. They have a measurement count, a hole count, an outlier count, and a $\chi^2$ sum. The track container is generic and allows experiments to add additional information through dynamic columns. An immediate benefit of this approach is that the branch stopper can now accumulate experiment specific statistics about the track, for instance the number of pixel subsystem holes, and make a more informed decision about whether to keep or drop the branch.

\section{Branch finalization}

The branch finalization is triggered when a propagation abort condition is met or a branch is stopped by the user. The current branch is then stored in the output track container, given it is selected to be kept. If other branches are still active, the CKF will continue with the next one, otherwise the CKF will stop.

This act of finalizing a branch was done in multiple places and essentially duplicated. This has been refactored to a single function which can be called from different places. This makes the code more maintainable and easier to reason about.
