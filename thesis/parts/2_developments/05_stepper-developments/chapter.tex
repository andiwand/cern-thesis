The stepper component plays a central role in the propagation of track parameters. In an inhomogeneous magnetic field, the trajectory of a chared particle cannot be expressed analytically. Therefore the equations of motions need to be integrated numerically in the general case, which the stepper is responisble for. Apart from that, it handles the transport of the Jacobian matrix and the covariance matrix of the track parameters. These are used in the Kalman filter to fit tracks to measurements. As such, the stepper is a crucial component for track reconstruction as a whole.

Current stepper implementations in ACTS are based on a fourth-order Runge-Kutta-Nystrom integration, which is described in Section \ref{sec:into/reco/runge-kutta-nystrom}. While this is well established for track reconstruction, there is not a single best way to implement it and to achieve optimal computational performance. Since the stepper is a performance critical component, developments have continued during the work on this thesis and will continue in the future.

This chapter dicusses the developments on the stepper component in ACTS during the work on this thesis. First, work on the \texttt{ConstrainedStep} is presented, which is a central helper class to deal with different step length requirements, followed by a discussion on the adaptive step size technique, which is used to improve the performance of the numerical integration. Then, improvements on the \texttt{EigenStepper} are discussed which is a long standing stepper implementation in ACTS. Afterwards, the \texttt{SympyStepper} is introduced which is a novel stepper implementation in ACTS, conceptualized and implemented as part of this thesis. Finally, the computational cost of the time parameter is discussed.

\section{Reworking the \texttt{ConstrainedStep}}

\texttt{ConstrainedStep} is a helper class in ACTS to deal with different step length requirements. Currently, ACTS deals with 4 different types: \texttt{accuracy}, \texttt{navigator}, \texttt{actor}, and \texttt{user}. The \texttt{accuracy} limit is used to communicate an appropriate step length limit to achieve the requested numerical integration accuracy target set by prior integration steps. The \texttt{navigator} limit holds the approximated distance to the next surface target. In case of overstepping the target, this can also be negative. The \texttt{actor} limit holds the approximate distance to the next abort condition, for example a trajectory path length limit or a target surface distance. Similar to the \texttt{navigator} limit, this can also be negative. The \texttt{user} limit is a user defined limit to restrict the step size to a given maximum.

\texttt{ConstrainedStep} takes inputs from the different sources and provides a single step length. The step length is determined by the minimum of the \texttt{accuracy}, \texttt{navigator}, \texttt{actor}, and \texttt{user} limits, while overstepping resolution is always preferred over smaller absolute distances.

This class used to be quite involved and a central cause of problems for the propagation. It has been effectively rewritten during the work on this thesis, to be more robust and easier to understand. One major compilcation was the handling of the step direction inside the \texttt{ConstrainedStep}. The propagation can be done in both directions, forward and backward. During a backward propagation we might overstep a target. To resolve this we need to reverse direction, effectively stepping forward. This was all handled inside the \texttt{ConstrainedStep} and resulted in failures as the signs where not treated correctly. In the new implementation this was simplified by removing the propagation direction sign from \texttt{ConstrainedStep}. With this change, it only deals with distances that have the propagation direction factored in. This is possible because the behavior of the \texttt{ConstrainedStep} has to be symmetrical for forward and backward propagation.

Another improvement was removing the sign of the \texttt{accuracy} limit. In rare cases the accuracy was set during overstep resolution which caused it to be negative. This effectively locked the step size resulting in a failed propagation due to reaching the step count limit. The new implementation decides the sign for the accuracy limit based on the \texttt{navigator} and \texttt{actor} limits.

\section{Implementation of an adaptive step size}
\label{sec:dev/stepper/adaptive-step-size}

Adaptive step size is a technique to improve the performance of the numerical integration given a certain accuracy target. The idea is to adjust the step size based on the local curvature of the trajectory. In regions of high curvature a smaller step size is used to resolve the trajectory more accuratly. In regions of low curvature a larger step size is used to accelerate the propagation and to limit the accumulation of machine precision errors.

This is done by computing an error estimate after each step of the numerical integration and comparing it to the requested accuracy target. If the error estimate is larger than the requested accuracy target the step size is reduced, otherwise it is increased.

A derivation of the expected error for the fourth-order Runge-Kutte-Nystrom is provided in \textcite{pub-2009-001} and only summarized here. The error estimate is given by
\begin{equation}
    \epsilon = h^2 \, \left\lVert \mathbf{k}_1 - \mathbf{k}_2 - \mathbf{k}_3 + \mathbf{k}_4 \right\rVert
\end{equation}
with the intermediate steps of the Runge-Kutta-Nystrom integration $\mathbf{k}_1$, $\mathbf{k}_2$, $\mathbf{k}_3$, and $\mathbf{k}_4$. Given the requested accuracy target $\tau$ and the used step length, the next step length can be calculated with \cite{pub-2009-001}\cite{numerical-recipes}:
\begin{equation}
    h_{\text{next}} = h \, \left( \frac{\tau}{\epsilon} \right)^{1/4} \text{,}
    \label{eq:dev/stepper/adaptive-step-size}
\end{equation}
where the exponent of 1/4 is determined by the order of the Runge-Kutta-Nystrom integration.

Equation \ref{eq:dev/stepper/adaptive-step-size} is used after each step of the numerical integration to adjuste the step size. If the error estimate is too large, the step size is reduced and the step is reattempted. If the error estimate is too small, the step size is increased to accelerate the next integration step.

Since the error estimate can fluctuate, an upper and lower limit is imposed to the updated step size:
\begin{equation}
    \frac{1}{4} h < h_{\text{next}} < 4 h \text{.}
\end{equation}

The accuracy condition is the following:
\begin{equation}
    \epsilon < 4 \tau \text{.}
\end{equation}
If not satisfied, the step is rejected and reattempted. The factor 4 provides some margin for the requested accuracy target and allows $\epsilon$ to fluctuate around $\tau$ without immediately rejecting steps.

The adaptive step size technique was implemented in ACTS during the work on this thesis and tested with the \texttt{EigenStepper}.

Figure \ref{fig:dev/stepper/adaptive-step-size} shows the error estimate and the step size as a function of integration steps with different transverse momenta in a 2 T magnetic field propagated with $\theta=\pi/4$ for 10 m. \section{Improvements to the \texttt{EigenStepper}}
\label{sec:dev/stepper/eigen}

The \texttt{EigenStepper} is a long standing stepper implementation in ACTS with the goal of being readable, maintainable and fast. It uses the Eigen library \cite{eigenweb} for linear algebra operations which is the de facto standard with C++. The math was once derived by hand and then transcribed into code.

The \texttt{EigenStepper} has an extension mechanism, mainly to deal with vacuum and dense material stepping, but also to generalize for potential future applications. Extensions are collected in a list and can bid for their priority in a given environment. An Auctioneer decides which extension is used for the current step based on the priority. The choosen extension is then used to calculate the intermediate steps of the Runge-Kutta-Nystrom integration and, in case of covariance transport, the transport Jacobian.

While the \texttt{EigenStepper} implementation is arguably doing well in terms of readability and maintainability, the extension mechanism diminishes these qualities. The Runge-Kutta-Nystrom integration is scattered between the stepper and the extensions which makes it hard to follow. To improve the situation, the \texttt{EigenStepper} was simplified during the work on this thesis. Instead of dealing with a list of extensions, only a single extension can now be provided. With this change, the dense extension explicitly falls back to the default extension, if no volume material is present. This simplification solves problems discovered during the work on the dense material propagation described in Chapter \ref{ch:dev/dense}, more specifically in Section \ref{sec:dev/dense/eigen}.

\section{Implementation of the \texttt{SympyStepper}}
\label{sec:dev/stepper/sympy}

The \texttt{SympyStepper} is a novel stepper implementation in ACTS, conceptualized and implemented as part of the work on this thesis. It is based on generated code from the symbolic math library \texttt{SymPy} \cite{sympy}. The motivation for this stepper is to improve CPU performance while keeping or improving the readable representation of the mathematics behind the implementation. Figure \ref{fig:dev/stepper/sympy} shows an example of using \texttt{SymPy} to generate C++ code for the first intermediate step of the Runge-Kutta-Nystrom integration.

\begin{figure}[ht]
    \centering
    \begin{subfigure}[t]{0.5\textwidth}
        \centering
        \begin{lstlisting}[language=Python,basicstyle=\tiny]
import sympy


l = sympy.Symbol("lambda", real=True)
d = sympy.MatrixSymbol("d", 3, 1)
B1 = sympy.MatrixSymbol("B1", 3, 1)

k1 = d.cross(l * B1)
        \end{lstlisting}
        \caption{Example Python code using \texttt{SymPy}}
    \end{subfigure}%
    ~
    \begin{subfigure}[t]{0.5\textwidth}
        \centering
        \begin{lstlisting}[language=C++,basicstyle=\tiny]
double x0 = B1[1] * d[2];
double x1 = B1[0] * d[2];
double x2 = B1[2] * d[0];
double x3 = B1[0] * d[1];
double k1[3];
k1[0] = lambda * (-x0 + B1[2] * d[1]);
k1[1] = lambda * (x1 - x2);
k1[2] = lambda * (-x3 + B1[1] * d[0]);
        \end{lstlisting}
        \caption{Generated C++ code}
    \end{subfigure}
    \caption{Example of using \texttt{SymPy} (a) to generate C++ code (b) for the first intermediate step of the Runge-Kutta-Nystrom integration.}
    \label{fig:dev/stepper/sympy}
\end{figure}

The \texttt{EigenStepper} uses the Eigen library for linear algebra operations with the goal of having readable and fast code, as the library is the de facto standard for linear algebra in C++ and supports vectorization. While this implementation has benefits in terms of readability and maintainability over the \texttt{AtlasStepper}, the extension mechanism diminishes these qualities. This is further discussed in Section \ref{sec:dev/dense/eigen}.

The \texttt{AtlasStepper} is based on the long standing Athena STEP algorithm and does not rely on any external libraries. The math operations are decomposed by hand and optimized for performance. This indeed results in fast code but is also practically unreadable and impossible to maintain. For example introducing a time parameter in the \texttt{AtlasStepper}, i.e. changing the matrix dimensions to handle, would be significantly more involved than in the \texttt{EigenStepper}.

Since maintainable is one of the key paradigms of ACTS, a stepper implementation cannot only focus on performance but should also be readable and extendable. How much performance can be sacrificed for readability and maintainability is a question each experiment has to answer for itself.

During the work on this thesis it was observed that the \texttt{EigenStepper} is about 2x slower than the \texttt{AtlasStepper}. At the same time this has a significant impact on the performance of the track finding procedure, which in turn is a performance critical algorithm for the track reconstruction for experiments, such as ATLAS. It was attempted to improve the performance of the \texttt{EigenStepper} but turned out to be quite difficult and at the cost of readability and maintainability.

The \texttt{SympyStepper} can be thought of as the midpoint between the \texttt{EigenStepper} and the \texttt{AtlasStepper}. The stepper math is written in Python using the \texttt{SymPy} library. This allows for high level linear algebra operations while also enabling the direct derivation of the linear error propagation from the Runge-Kutta-Nystrom integration. Apart from that, we can use common subexpression elimination to simplify the math upfront and reduce the number of operations. Another big advantage is that one can encode known values and symmetries from the beginning, which directly results in simplified terms and fewer operations. An example for known values are zeros and ones in a Jacobian. We can also make use of the symmetry of the covariance matrix to avoid duplicate work. These kinds of simplifications are not possible with Eigen. Finally, we can generate C++ code from the \texttt{SymPy} expressions which is then used in the ACTS stepper implementation. The generated code is not meant to be readable but fast. It can be thought of as binary code produced by a compiler and can be generated by the build system during the compilation of ACTS.

\input{parts/2_developments/05_stepper-developments/tables/sympy_perf.tex}

Table \ref{tab:dev/stepper/sympy-perf} shows the relative CPU time for the \texttt{EigenStepper}, AtlasStepper, and \texttt{SympyStepper} with and without Jacobian transport. The values are obtained with a syntetic benchmark where 1 GeV tracks are propagated through a 2 T field for a path length of 5 m. The \texttt{EigenStepper} is significantly slower than the AtlasStepper and the \texttt{SympyStepper}. The \texttt{SympyStepper} sits somewhere in the middle which means that there is still room for improvement for the \texttt{SympyStepper}. While the AtlasStepper is tuned by hand for performance, the \texttt{SympyStepper} relies on code generation which is not necessarily optimal for the hardware yet.

Another advantage of the AtlasStepper is that the Jacobian transport is not done in free to free but bound to free which reduces the matrix size and the number of operations. This was not attempted with the \texttt{EigenStepper} and the \texttt{SympyStepper} yet as the free to free might be needed by fitters which work with free parameters in the future. Since this is not the case yet one could make this functionallity switchable and potentially profit from the reduced matrix size and number of operations in the pure bound parameter case.

\section{The cost of the time parameter}

As mentioned in Section \ref{sec:intro/acts/track-parameterization}, the track parametrization is not customizable in ACTS. As such, the time parameter will always be present, even if it is not needed by experiments. This is for example the case for the ATLAS ITk detector, which does not provide timing information. Since track reconstruction is a performance critical task, and the time parameter requires additional operations and increases vector and matrix dimensions, the computational cost attached to it has to be evaluated.

In order to measure the performance of ACTS without a time parameter, it was consistently removed in a separate branch and compared with the performance obtained with the baseline implementation. This is mostly a mechanical process which at the time being involved changing 242 files with 1687 additions and 3281 deletions.

\input{parts/2_developments/05_stepper-developments/tables/time_param_perf.tex}

Table \ref{tab:dev/stepper/time-param-perf} shows the relative CPU time of the \texttt{EigenStepper} and the \texttt{SympyStepper} with and without time as an additional track parameter. These values were obtained with the OpenDataDetector, \ttbar events at $\sqrt{s} = 14 \ \text{TeV}$, $\langle \mu \rangle = 200$, using fast simulation and a standard reconstruction workflow. The relative CPU times are extracted from the track finding procedure, which is a performance critical component and uses the stepper heavily for the propagation of track parameters and their covariance. The proportion of CPU time spent in the stepper is 25\% for the \texttt{EigenStepper} with time and 14\% for the \texttt{SympyStepper} with and without time. This measurement shows that the time parameter is indeed costly, even if it is not used. In case of the \texttt{EigenStepper} we see a performance improvement of 33\% then the time parameter is removed which scales to 9\% for the track finding procedure. The \texttt{SympyStepper} is able to recover most of these losses and only shows a 8\% performance improvement when the time parameter is removed which scales to 1\% for the track finding procedure.

How the remaining 8\% in case of the \texttt{SympyStepper} can be recovered is under investigation. Potentially there are a few more places where the \texttt{SymPy} approach can be used to speed up the stepping. Alternatively, the \texttt{SympyStepper} could be adapted to only transport the time parameter if it is needed. One only needs to remove the time parameter from the stepper math, the additionally generated stepper code can then be used if the time parameter is not needed. The more involved architectural problem is how to communicate if the time parameter is needed or not, which is currently not possible in ACTS and requires changes in the EDM, especially if this should be generalized to all track parameters. For instance tracking without a magnetic field will not provide information about the momentum which could also be handled in a similar way.
